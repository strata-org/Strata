Problem Statement

Goal: Prove program transformation correctness in Lean 4 — each transform must overapproximate.

Target theorems:

  • detToKleene_overapproximates — deterministic → Kleene algebra (778 lines, 18 lemmas)
  • loopElim_overapproximatesAggressive — loop elimination (11,678 lines, 221 lemmas)

A human wrote both — but a single LLM agent cannot. Three failure modes:

  1. Context loss — agent forgets what it tried
  2. Cycles — sub-goals identical to ancestors
  3. No shared memory — sibling branches repeat dead ends

What Is "Overapproximates"?

A transform T : Stmt → Option Stmt overapproximates if:

Every reachable terminal/exiting state from st is also reachable from T(st), and assertion failures are preserved.

Intuitively: the transformed program can do at least everything the original can.

Simple Example — wrapCmdInBlock

def wrapCmdInBlock (s : Stmt P (Cmd P)) : Option (Stmt P (Cmd P)) :=
  match s with
  | .cmd c => if Cmd.definedVars c = [] then
                some (.block "wrapper" [.cmd c] .empty)
              else none
  | _ => none
theorem wrapCmdInBlock_overapproximates :
  Overapproximates wrapCmdInBlock

One-lemma proof — a single agent closes it directly.

v0: Let Agents Talk (failed)

Idea: Message-passing, agents self-organize.

  • Nobody tracked the big picture
  • No priority — worked on whatever came next
  • Two writers hit same dead-end independently
  • Failures not shared across agents

Result: Activity without convergence.

v1: Recursive Tree + Sandboxing (better)

Idea: Proof as a tree of independent sub-problems. Hard permission boundaries.

Still failed: No coordination on what next, when to stop, cycle detection.

v2: The Orchestration Layer

Tree assumption wrong — sub-lemmas share helpers, form cycles. It's a DAG.

A deterministic state machine that:

  • Models obligations as a DAG with shared deps + reuse
  • Global ledger of all obligations + status
  • Priority: hardest/most generic first
  • Cycle detection across entire DAG
  • Prunes dead branches, re-activates stuck lemmas

Design: Never-delete DAG. BM25 search. Indegree priority. JSON persistence. Read-only MCP.

Cycle detection — three layers:

  1. Fast: Signature hash vs ancestry — O(1)
  2. Soft: LLM catches semantic aliases
  3. Hard: Lean compiler — exact/apply ancestor? → confirmed

Why This Might Work

LLMs already prove individual goals. Same principles scale to orchestration:

COPRA (single goal)Our system (multi-file)
Try tactic, observe new goalTry proof, observe sorries
Remember failed tacticsLedger remembers failed branches
Detect loops (goal ≥ ancestor)Detect cycles (sig ≈ ancestor)
Retrieve relevant lemmasSearch ledger for proved helpers
Backtrack on failurePrune + re-activate parent

Insight: Orchestration feeds the right goal with the right context — stops when cycling.

Thank You — Questions?

Key takeaway: If an LLM can close a single goal given the right context, then orchestration's job is to keep feeding it the right goal with the right context — and stop when it's cycling.

  • A persistent ledger that never forgets what failed
  • Cycle detection that catches semantic equivalence, not just text matches
  • Priority-driven focus: depth with memory > breadth without

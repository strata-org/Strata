# Paren-strip iterative fix on `elabExpr` — experimental results

**Date:** 2026-06-02
**Strata commit (origin/main):** `75f005566eb405b774dd799d5a138faeea6e008f`
**Issue under test:** [strata-verify-stack-overflow-deeply-nested-expr.md](./strata-verify-stack-overflow-deeply-nested-expr.md) (referred to as "issue 2" in our reports)
**Verdict:** Insufficient. Discard.

## Hypothesis tested

The recursive `Init.exprParen` arm in `StrataDDM/StrataDDM/Elab/Core.lean:1696` (`elabExpr tctx stx[1]`) consumes one OS thread stack frame per layer of paren wrapping. Replacing this self-recursion with a `while` loop should reduce stack consumption on deeply parenthesised inputs and shift the threshold at which `Stack overflow detected. Aborting.` fires.

## What was changed

One edit in `StrataDDM/StrataDDM/Elab/Core.lean` at `elabExpr` (line 1694). The original `match` arm:

```lean
partial def elabExpr (tctx : TypingContext) (stx : Syntax) : ElabM Tree := do
  match stx.getKind with
  | `Init.exprParen =>
    elabExpr tctx stx[1]
  | `Init.exprIdent =>
  ...
```

was replaced with iterative paren stripping:

```lean
partial def elabExpr (tctx : TypingContext) (stx : Syntax) : ElabM Tree := do
  let mut stx := stx
  while stx.getKind = `Init.exprParen do
    stx := stx[1]
  match stx.getKind with
  | `Init.exprIdent =>
  ...
```

Net diff: ~9 lines added, 2 lines removed. Build succeeds (`lake build strata` from `StrataCLI/`).

## Inputs


## Procedure

For each input, ran `strata verify --parse-only <file>` under a 30 s `gtimeout` and recorded:

- **rc**: 0 = clean exit, 124 = wall-clock timeout, 134 = SIGABRT (stack overflow)
- **elapsed**: wall-clock from spawn to exit
- **out**: first line of stdout (`Successfully parsed.` if elaboration finished, blank otherwise)

Used `--parse-only` rather than `--check-mode deductive` because the SIGABRT happens during DDM elaboration (before the verify pipeline starts), so this isolates the bug under test from downstream concerns.

Pre-fix and post-fix binaries were both built from the same `origin/main` worktree (`/tmp/draft-fix-elabexpr-paren-1780395800/`), differing only in the one-edit patch above. Pre-fix binary obtained by `git stash` + rebuild, then `git stash pop`.

A pre-fix run on N=8,000 returned `rc=124 elapsed=15,023 ms` (timeout, no SIGABRT). Post-fix on the same input also timed out without SIGABRT. So the paren-strip fix did *not* enable a new threshold here either — both versions choke on parser slowness before hitting `elabExpr`'s paren recursion. At N≥20,000 the SIGABRT fires (in *parsing*, not in `elabExpr`'s paren arm — confirmed because the failure mode is identical pre- and post-fix on this input shape).

## Interpretation

The fix is mechanically correct (Lean accepts the loop, the binary builds, no test regressions observed in spot-checks) but **the paren arm is not the dominant frame consumer for the documented reproducer**. Per ITE level on Family 1 inputs, frame consumption decomposes roughly as:

| Source | Frames per level (estimate) |
|---|---:|
| Outer paren wrap (`(if ...)`) | 1 (eliminated by fix) |
| `Core.if` op-elaboration via `elabExpr → runSyntaxElaborator → elabSyntaxArg → elabExpr` | 3 (untouched) |
| Cond paren wrap (`(x == N)`) | 1, but cond is shallow so only counts on one descent — eliminated by fix |
| Then-branch paren wrap | 1 (eliminated by fix) |

So the fix eliminates 1 frame per ITE level on the deep path (the then-branch paren), or possibly 2 if the parser exposes the cond's paren on every descent. The remaining ~3 frames per level from the operator-elaboration chain dominate.

To shift the threshold for the documented reproducer, the operator-elaboration chain must also be made iterative. That requires CPS-converting `elabExpr`, `runSyntaxElaborator`, and `elabSyntaxArg` together — a much larger refactor in a critical mutual block, not a one-line surgical fix.

## Side observation: parser slowness independent of stack overflow

The N=2000 / N=3000 / N=4000 ITE inputs (Family 1) and N=4000 / N=8000 pure-paren inputs (Family 2) **time out at 30 s without SIGABRT**, both pre-fix and post-fix. This is a different failure mode than the documented stack overflow. It suggests an O(N²) (or worse) cost in parsing or DDM elaboration on these input shapes, which is unrelated to the bug under test. Worth tracking as a separate issue.

## Conclusion

The paren-strip fix is **discarded**. It does not move the threshold for the documented reproducer, the ratio of effort-to-effect is poor, and the deeper fix (full trampoline of the elabExpr/runSyntaxElaborator/elabSyntaxArg cycle) does not benefit from this micro-change as a stepping stone — it would replace the entire elabExpr body anyway.

Worktree at `/tmp/draft-fix-elabexpr-paren-1780395800` and branch `draft-fix/elabexpr-paren` to be removed; nothing was committed. The reproducer files at `/tmp/r-*.core.st` are kept for re-testing if a future fix attempt needs a baseline.

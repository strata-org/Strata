## Response to Comment 1 (ASTtoCST.lean:422 — SNegOverflow mapped to neg_expr)

Fixed in 0f524cc. `SNegOverflow` and `UNegOverflow` are now mapped to `Bool.Not` instead of `neg_expr`, which is type-correct (boolean predicate, not arithmetic negation). These overflow predicates only appear in PrecondElim-generated assertions, not in user-written code, so the CST representation is only used for debug printing.

## Response to Comment 2 (ASTtoCST.lean:465 — Missing overflow predicate entries)

Fixed in 0f524cc. All binary overflow predicates (`SAddOverflow`, `SSubOverflow`, `SMulOverflow`, `SDivOverflow`, `UAddOverflow`, `USubOverflow`, `UMulOverflow`) are now in `bvBinaryOpMap` with a boolean CST approximation. They no longer fall through to the error path.

## Response to Comment 3 (PrecondElim.lean:60 — SafeSDiv/SafeSMod misclassification)

Fixed in 0f524cc. `classifyPrecondition` now takes a `precondIdx` parameter, and `collectPrecondAsserts` tracks the per-function precondition index. For `SafeSDiv`/`SafeSMod`, precondition 0 (y ≠ 0) is classified as `divisionByZero`, and precondition 1 (¬SDivOverflow) is classified as `arithmeticOverflow`.

## Response to Comment 4 (Options.lean:90 — unsignedOverflowCheck not wired)

Addressed in 0f524cc. Added documentation explaining that this is forward-looking infrastructure: the option will be wired when a front-end language (Laurel or a future C front-end) gains bitvector types. Currently no front-end emits BV operations, so there's nothing to wire it to yet. The option documents the design intent that unsigned overflow checking should be opt-in.

## Response to Comment 5 (Factory.lean:343 — Float64 safe ops have no preconditions)

Fixed in 0f524cc. `Float64.SafeAdd/Sub/Mul/Div` now carry preconditions requiring `¬Float64.*Overflow(x, y)`. Each is constructed as a `WFLFunc` with an explicit precondition expression referencing the corresponding overflow predicate function. `PrecondElim` now generates overflow VCs at every Float64 safe arithmetic use site.
